# 2.1 Tabular vs Sequential 데이터 차이

> **이 토픽의 목표**
> "왜 현재 프로젝트의 `(2310, 17)` XGBoost 입력 구조를 `(N, T, 17)` 시퀀스로 바꿔야 하는가"를 통계적 근거 + 데이터 구조 관점에서 설명할 수 있게 됩니다.
> 1.3(자기상관)과 1.4(분해) 에서 관측한 MSFT 실측치를 **동기**로 연결합니다.

---

## 0. 한 줄 요약

- **Tabular**: 행 한 개 = 관측치 한 개. `(N, F)` 모양. 행끼리 **독립**이라고 가정.
- **Sequential**: 샘플 한 개 = 과거 T일치 관측치 묶음. `(N', T, F)` 모양. 시간 의존성을 **명시적으로** 보존.
- 금융 수익률처럼 **자기상관이 살아있는 데이터**에서는 tabular 가정이 깨지므로 sequential 입력이 필요합니다.

---

## 1. 현재 프로젝트 구조부터 확인

`black_litterman/Test/Step3_IT_WalkForward_panel.ipynb` 의 설정:

```python
FEATURE_COLS = [
    "log_return_1d", "simple_return_1d",
    "mom_1m", "mom_3m", "mom_6m", "mom_12m", "mom_12m_skip_1m",
    "vol_20d_ann", "vol_60d_ann", "vol_252d_ann",
    "mkt_rf", "smb", "hml", "rmw", "cma", "rf", "mom_factor",
]  # 총 17개
TARGET_COL   = "fwd_ret_21d"
IS_DAYS      = 252  # 1년치 (학습 구간)
IT_TICKERS   = ["MSFT", "INTC", ... ]  # 10 종목
```

한 번의 walk-forward 학습에서 XGBoost가 받는 `X_is` 의 shape 는

```
(IS_DAYS × n_ticker, n_feature)
= (252 × 10, 17)   # 실제 코드에선 유효 샘플만 남기므로 2310 부근
```

**이 배열은 완전히 tabular입니다.**
각 행은 "어느 날 어느 종목의 17개 지표"일 뿐, 그 행이 시계열의 몇 번째 시점인지 XGBoost는 신경 쓰지 않습니다.

---

## 2. Tabular 의 본질 — "행은 독립이다"

Tabular 가정의 수학적 표현:

$$
(x_i, y_i) \stackrel{\text{i.i.d.}}{\sim} P(X, Y), \quad i = 1, \dots, N
$$

모든 행이 **동일한 분포에서 독립적으로 뽑혔다**고 봅니다.
트리 계열 모델(XGBoost·LightGBM)은 이 가정 아래 설계되었고, 구체적으로는

- 행 단위 샘플링(bootstrap, subsample)을 자유롭게 허용
- 분할 기준을 피처 값 그 자체로만 계산
- 어떤 행이 학습 집합에 있든 예측 성능은 행의 "순서"와 무관하게 같아야 함

**문제:** 금융 수익률은 i.i.d. 가 아닙니다.
1.3 실습에서 MSFT 에 대해 관측한 것:

| 대상 | lag 1 자기상관 | Ljung-Box p-value | 해석 |
|---|---|---|---|
| `log_return_1d` (수익률) | −0.099 | 10⁻⁸ 수준 | 평균회귀 존재 |
| `sq_log_return_1d` (제곱) | +0.357 | 10⁻²⁸⁷ 수준 | 변동성 클러스터링 극심 |
| `abs_log_return_1d` (절댓값) | +0.227 | 10⁻¹⁸⁰ 수준 | 큰 변동 뒤 큰 변동 |

- 수익률 자체엔 약한 평균회귀
- **변동성 대리변수**(제곱·절댓값)엔 매우 강한 양의 자기상관

즉 "어제 있었던 일"이 "오늘 일어날 일"의 분포에 영향을 줍니다.
Tabular 가정(행 독립)은 틀렸습니다.

---

## 3. Sequential 의 본질 — "차원이 하나 더"

Sequential 입력은 한 샘플을 **시간축을 포함한 2차원 블록**으로 봅니다.

```
Tabular 샘플 1개     :   x_i ∈ ℝ^F                    shape (F,)
Sequential 샘플 1개  :   X_i ∈ ℝ^{T × F}              shape (T, F)
```

한 샘플에 **과거 T일치**의 F개 피처가 통째로 들어갑니다.
배치 차원까지 포함하면

```
Sequential 입력      :   shape (N', T, F)
                                │   │   │
                                │   │   └── 피처 수
                                │   └────── 시퀀스 길이 (lookback window)
                                └────────── 샘플 수
```

`N'` 은 원래 N에서 `T-1` 만큼 줄어듭니다. 처음 T-1 개 시점은 "과거 T일이 부족해서" 샘플을 만들 수 없기 때문입니다.

---

## 4. 왜 이 구조가 필요한가 — 1.3/1.4 근거

### 4.1 1.3에서 본 것: 자기상관은 죽지 않는다

변동성 대리변수의 ACF가 lag 1~20까지 양수로 깔리며 Ljung-Box p ≈ 0. 이는
> "오늘의 변동성 크기는 어제의 변동성 크기와 독립이 아니다"

는 말입니다. Tabular 모델로 이 패턴을 학습하려면

1. "어제 log_return", "그저께 log_return", ... 를 **수동 피처로 전개**하거나
2. "과거 20일 표준편차" 같은 **집계 피처**를 만들어 넣어야 합니다.

현재 프로젝트의 `vol_20d_ann`, `vol_60d_ann`, `vol_252d_ann` 이 바로 2번 시도입니다.
그러나 이는 **평균적인 변동성**만 담지, "지난 5일 동안 변동성이 빠르게 상승하고 있는 중"인지 여부는 표현하지 못합니다.

Sequential 모델(GRU/LSTM)은 T일치 원본 시퀀스를 그대로 받아 이런 **동적 패턴**을 자체적으로 학습합니다.

### 4.2 1.4에서 본 것: 고전 분해가 안 먹힌다

1.4 실습에서 확인한 MSFT 결과:
- `seasonal_decompose` 잔차 Ljung-Box p ≈ 0
- STL 잔차도 p ≈ 0
- → 분해 후에도 **자기상관이 여전히 남음**

다시 말해, 단순한 추세·계절성 모델로는 수익률의 시간 구조를 설명할 수 없고, 고급 시퀀스 모델이 필요하다는 뜻입니다.

---

## 5. 피처 엔지니어링으로 우회할 수는 없나?

현실적으론 일부 가능하지만 한계가 뚜렷합니다.

### 5.1 lag 피처 전개

```python
df["ret_lag_1"]  = df["log_return_1d"].shift(1)
df["ret_lag_2"]  = df["log_return_1d"].shift(2)
...
df["ret_lag_60"] = df["log_return_1d"].shift(60)
```

이 방식으로 60일을 쌓으면 피처가 `17 + 60*17 = 1037` 개로 폭증합니다.
- 트리 모델은 피처 수가 많아지면 과적합 위험 증가
- 시점 간 상관이 매우 높아 트리 분할이 **어느 lag를 고를지** 흔들림
- 결정적 한계: **lag 간 관계**(1·2일 전이 같이 올라갔다는 "추세성")를 표현하려면 추가로 "1·2일 전의 곱", "1·2일 전의 합" 같은 조합 피처가 필요

### 5.2 윈도우 집계 피처

```python
df["ret_rolling_5"]   = df["log_return_1d"].rolling(5).mean()
df["ret_rolling_20"]  = df["log_return_1d"].rolling(20).mean()
df["vol_rolling_20"]  = df["log_return_1d"].rolling(20).std()
```

`vol_20d_ann` 등이 이 범주입니다. 평균·표준편차 같은 **1·2차 모먼트 요약**은 담지만,
- "상승 → 하락 전환점이 며칠 전에 있었는가"
- "직전 3일 연속 고변동성"
같은 **형상 정보**는 잃어버립니다.

### 5.3 Sequential 모델이 하는 일

GRU는 내부적으로 **학습 가능한 가중치 벡터로 T일치 입력을 요약**합니다.
- `ret_rolling_20` 처럼 사람이 미리 정한 "평균"이 아니라,
- 데이터에 맞춰 **어떤 시점에 어떤 가중치를 두어야 할지** 자동으로 찾습니다.

> 피처 엔지니어링 = 사람이 시퀀스를 고정 함수로 요약
> 시퀀스 모델      = 모델이 시퀀스를 학습 가능한 함수로 요약

## 6. 트레이드오프

Sequential 구조로 바꾸면 얻는 것만 있지 않습니다.

| 항목 | Tabular(XGBoost) | Sequential(GRU) |
|---|---|---|
| 샘플 수 | N (전부 사용) | N − T + 1 (T=60이면 앞 59일 손실) |
| 피처 수 | F | 같은 F를 T번 보여줌 (파라미터는 모델 내부에) |
| 학습 속도 | 매우 빠름 (CPU) | 느림 (GPU 권장) |
| 해석 가능성 | 변수 중요도(SHAP 등) 쉬움 | 시점·피처 기여도 해석 어려움 |
| 소량 데이터 대응 | 강함 | 약함 (과적합 위험) |
| 시간 의존성 | 수동 피처에 의존 | 자동 학습 |
| 비정상성 | 트리 분할로 일부 대응 | 학습 구간 외 패턴엔 취약 |

### 6.1 샘플 수 감소의 의미

현재 walk-forward 설정:
- IS 구간 252일 × 10종목 = 2520 행
- Sequential(T=60)로 변환 → (252 − 60 + 1) × 10 = **1930 샘플**
- 약 **23%** 샘플 손실

손실된 59일은 "Sequential 샘플을 만들기엔 부족하지만 lookback 재료로는 쓰이는" 기간이라 완전히 낭비되진 않습니다. 다만 학습 집합이 작아지는 건 사실이라, T 값을 고를 때 이 트레이드오프가 첫 번째 고려 사항입니다.

## 7. 이 구조 전환이 Black-Litterman 단계에 미치는 영향

현재 파이프라인 4단계에서 **3단계만** 변합니다.

```
1. universe 선정           ← 변경 없음
2. feature build           ← 변경 없음 (동일 17개 피처)
3. Walk-forward 예측 모델   ← XGBoost → GRU (이번 학습의 목표)
   ├─ 입력 shape           ← (2310, 17) → (N', 60, 17)
   ├─ 레이블                 ← fwd_ret_21d → 5분위 one-hot (동일)
   └─ 출력                   ← proba shape (N', 5) (동일)
4. Black-Litterman 최적화   ← 변경 없음 (proba만 받으면 됨)
```

**핵심:** 3단계의 **입출력 인터페이스**는 유지하면서 **모델 내부**와 **입력 shape**만 바꿉니다.
이 때문에 `fit → predict_proba` 를 감싸는 얇은 어댑터(`SequenceClassifierAdapter`) 정도만 새로 쓰면 되고, walk-forward/BL 쪽 코드는 그대로 돌릴 수 있습니다.

## 부록 A — 용어 정리

- **Lookback window (T)**: 한 샘플을 만들 때 과거 몇 일치 데이터를 사용할지. 이번 프로젝트에서는 T=60 근방을 후보로 검토합니다.
- **Prediction horizon (h)**: 레이블이 몇 일 뒤의 수익률인지. 현재 프로젝트에선 `fwd_ret_21d` 이므로 h=21.
- **Sequence length**: 보통 T와 같은 의미로 쓰입니다.
- **Look-ahead bias**: 학습 시점보다 미래의 정보가 입력으로 들어가는 오류. 2.3에서 집중적으로 다룹니다.
- **Batch size**: 한 번에 모델에 흘려 보내는 샘플 수. Sequential 입력에서 배치 차원이 제일 앞에 옵니다.

## 부록 B — 왜 "T일치 블록을 겹치게" 쌓는가

일반적인 방법은 **stride=1 sliding window**입니다.

```
t=0:  [row_0, row_1, ..., row_59]  → sample 0
t=1:  [row_1, row_2, ..., row_60]  → sample 1
t=2:  [row_2, row_3, ..., row_61]  → sample 2
...
```

겹치게 뽑으면 샘플 수가 최대로 확보되지만, 인접 샘플 간 정보가 중복되어 **사실상 독립 표본 수는 적어집니다**. walk-forward 검증에서 OOS 구간을 얇게 잡을 때 이 중복이 왜곡을 키울 수 있어, 2.3/2.5에서 다시 논의합니다.
