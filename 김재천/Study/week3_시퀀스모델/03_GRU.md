## 1. 왜 GRU인가 — 탄생 배경

**2014년**, Cho et al.이 기계번역(NMT) 논문 *"Learning Phrase Representations using RNN Encoder-Decoder"* 에서 제안했습니다. 핵심 동기는 단순합니다.

> "LSTM은 효과는 좋은데 게이트 3개, 파라미터 많고, 직관적으로 왜 저 구조인지 설명하기 어렵다. 더 단순하게 같은 일을 할 수 있지 않을까?"

LSTM이 해결한 문제는 "**과거를 오래 기억하고, 필요할 때 꺼내 쓰기**"였습니다. GRU는 이걸 **두 개의 게이트**만으로 해냅니다.

| | LSTM (1997) | GRU (2014) |
|---|---|---|
| 게이트 수 | 3개 (f, i, o) + 후보 g | 2개 (r, z) + 후보 h̃ |
| 상태 | c_t (장부) + h_t (게시판) 분리 | h_t 하나로 통합 |
| 파라미터 | 4·(H² + HF + H) | 3·(H² + HF + H) |
| 학습 속도 | 기준 | LSTM 대비 ~25% 빠름 |
| 성능 | 기준 | 대부분 과제에서 LSTM과 비슷 |

**핵심 메시지**: GRU는 LSTM의 "기억 유지" 본질만 뽑아서 간소화한 버전입니다. 덜어낸 것이 많음에도 성능 손실은 크지 않아서, **실무에서 자주 기본 선택지**가 됩니다.

## 2. GRU의 핵심 구조 — 게이트 2개, 상태 1개

LSTM과의 가장 큰 구조적 차이를 한 장으로 요약하면:

```
LSTM:
  과거 ──┬─► [forget gate f] ──► c_t (장부)
         │
  입력 ──┼─► [input gate i] ──► g (후보) ──► c_t 에 더함
         │
         └─► [output gate o] ──► h_t (게시판) ◄── tanh(c_t) 필터링

GRU:
  과거 ──┬─► [reset gate r] ──► h̃ (후보) 계산 시 과거를 얼마나 볼지
         │
  입력 ──┤
         │
         └─► [update gate z] ──► h_t = (1-z)·h_{t-1} + z·h̃  (섞기)
```

**GRU의 철학**:
- "장부"(c)와 "게시판"(h)을 굳이 분리하지 말고 **하나의 상태 h로 통합**하자.
- "잊기"(forget)와 "더하기"(input)도 **하나의 update gate로 묶어서** 서로 반대 관계(1-z ↔ z)로 표현하자.
- "출력 필터링"(output gate)은 **생략**하자 — 상태 자체가 곧 출력.

→ 게이트 수: 3개 → 2개. 파라미터: 4블록 → 3블록. 그래도 **기억/망각 기능은 그대로 유지**.

## 3. 수식 4줄로 본 GRU 전체

```
r_t = σ(W_r · [h_{t-1}, x_t] + b_r)                ── (1) reset gate
z_t = σ(W_z · [h_{t-1}, x_t] + b_z)                ── (2) update gate
h̃_t = tanh(W_h · [r_t ⊙ h_{t-1}, x_t] + b_h)      ── (3) 후보 hidden
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t            ── (4) 새 hidden
```

**각 수식이 하는 일**:
- (1) **r_t**: "후보를 만들 때 과거를 얼마나 참고할까" (0=완전 무시, 1=완전 반영)
- (2) **z_t**: "최종 상태에서 새 것을 얼마나 채택할까" (0=과거 유지, 1=새 것으로 교체)
- (3) **h̃_t**: "이 시점의 후보 상태" — 과거를 r로 게이팅한 뒤 입력과 섞어서 만듦
- (4) **h_t**: "보간" — 과거와 후보를 z 비율로 섞음

### 기호 주의

- σ = sigmoid (값 0~1, 게이팅 비율)
- tanh = 쌍곡탄젠트 (값 -1~+1, 실제 내용)
- ⊙ = Hadamard product (요소별 곱)
- · = 행렬곱
- [a, b] = 벡터 concat

### LSTM 수식과의 매핑

| LSTM | GRU | 의미 |
|---|---|---|
| `f_t ⊙ c_{t-1}` | `(1−z_t) ⊙ h_{t-1}` | 과거 유지 |
| `i_t ⊙ g_t` | `z_t ⊙ h̃_t` | 새 내용 추가 |
| `o_t ⊙ tanh(c_t)` | (없음) | GRU는 출력 필터링 생략 |
| `c_t` vs `h_t` 분리 | `h_t` 하나 | 상태 통합 |

**핵심 관찰**: LSTM의 `c_t = f·c + i·g`에서 f와 i가 독립이었지만, GRU는 `z`와 `(1-z)` 로 **한 파라미터로 양쪽을 동시에 결정**합니다. "과거를 80% 유지" = "새 것을 20%만 반영"이 자동으로 강제됩니다.

---

## 4. Reset Gate r_t — "과거를 얼마나 무시할까"

### 역할

**후보 상태(h̃_t)를 만들 때**, 이전 hidden이 얼마나 개입할지 결정합니다.

```
r_t = σ(W_r · [h_{t-1}, x_t] + b_r)   ∈ (0, 1)
h̃_t = tanh(W_h · [r_t ⊙ h_{t-1}, x_t] + b_h)
              ↑
              r_t가 h_{t-1}을 게이팅
```

- **r_t ≈ 1** → "과거를 그대로 반영해서 후보 생성" (연속적 상태)
- **r_t ≈ 0** → "과거를 거의 무시하고 현재 입력만으로 후보 생성" (리셋)

### 비유: "초안 쓸 때 과거 회의록 참고 비율"

회의 후 새 회의록을 쓴다고 합시다. r_t는 **"지난 회의록을 얼마나 참고할지"** 의 다이얼입니다.
- r=1: "지난 회의록 그대로 보면서 이어서 쓰기" → 연속성 유지
- r=0: "지난 회의록 덮고 지금 이야기만으로 새로 쓰기" → 맥락 단절

### 왜 r_t가 필요한가

극단적 예: 문장의 끝이면 "지금까지 읽은 내용은 잠시 잊고 다음 문장을 새로 시작" 해야 할 때가 있습니다. r_t가 0에 가까워지면 네트워크는 이렇게 "**부분 리셋**"을 할 수 있습니다.

### 수학적 영향

r_t = 0이면 후보 h̃_t가 입력 x_t만의 함수가 됩니다:
```
h̃_t = tanh(W_h · [0, x_t] + b_h) = tanh(W_hx · x_t + b_h)
```
과거 의존성이 사라지는 효과 → 새 chunk 시작.

---

## 5. Update Gate z_t — "새 것과 옛 것을 몇 대 몇으로 섞을까"

### 역할

**최종 hidden 상태**에서 과거(h_{t-1})와 새 후보(h̃_t)의 **혼합 비율**을 결정합니다.

```
z_t = σ(W_z · [h_{t-1}, x_t] + b_z)   ∈ (0, 1)
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t
         ↑                    ↑
      과거 유지 비율        새 것 채택 비율
```

- **z_t ≈ 0** → "과거(h_{t-1})를 거의 그대로 복사" (변화 없음)
- **z_t ≈ 1** → "새 후보(h̃_t)로 완전히 교체"

### 비유: "이번 주 보고서 업데이트 비율"

매주 내놓는 회사 현황 보고서를 생각해봅시다. z_t는 **"이번 주에 얼마나 새로 쓸지"** 의 비율입니다.
- z=0: "지난 주 보고서 그대로 제출" (경영진 입장에선 "아무 일 없었음")
- z=1: "지난 주 보고서 다 버리고 새로 작성" (완전히 새 페이즈 진입)
- z=0.3: "70%는 지난 주 유지, 30%만 새 내용 추가" (점진적 업데이트)

### LSTM의 forget·input gate와 비교

LSTM: `c_t = f·c_{t-1} + i·g` → f와 i가 **독립적**이라 둘 다 1이 되면 값이 폭발할 수 있음 (이래서 3.2의 cell state 그래프가 넓게 퍼진 것).

GRU: `h_t = (1-z)·h_{t-1} + z·h̃` → **합이 항상 1**이라 **볼록 결합(convex combination)**. 값이 한쪽의 범위(−1~+1)를 벗어나지 않음. 더 안정적.

### 왜 이게 중요한가 — "기본값이 과거 유지"

z_t 초기화를 생각해보면, 가중치가 작은 랜덤 값으로 시작하므로 `z_t ≈ σ(0) = 0.5` 근처에서 출발합니다. 만약 학습 초기에 z_t가 0.1 이하로 떨어지면 **"이번 타임스텝에 10%만 업데이트"** → 정보가 거의 그대로 흐릅니다. 이게 곧 **gradient highway** 역할을 합니다.

## 6. GRU도 gradient highway가 있는가

**있습니다.** 형태는 LSTM과 다르지만 본질은 같습니다.

### LSTM의 경우 (복습)

```
∂c_t / ∂c_{t-1} = f_t   (덧셈 구조 덕분)
```
→ f_t가 1에 가까우면 60시점 뒤에도 gradient가 살아남음.

### GRU의 경우

```
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t

∂h_t / ∂h_{t-1} ≈ (1 − z_t) · I + (다른 경로들)
                    ↑
                 dominant 경로
```

정확히 펼치면:
```
∂h_t / ∂h_{t-1} = diag(1 − z_t)                     ← 직접 경로
                 + diag(z_t) · ∂h̃_t/∂h_{t-1}         ← 후보 경로
                 + diag(h_{t-1} − h̃_t) · ∂z_t/∂h_{t-1}  ← z 변화 경로
```

**핵심은 첫 항**: `diag(1 − z_t)`. z_t가 0에 가까우면 **1에 가까운 대각행렬**이 돼서 gradient가 **거의 그대로 통과**합니다.

### "update gate bias를 음수로 초기화"하는 트릭

LSTM에서 `b_f = 1`로 초기화했던 것처럼, GRU에서는 **`b_z`를 음수로 초기화**하는 기법이 있습니다.
- `b_z = -2` → z_t의 평균이 σ(-2) ≈ 0.12
- → (1 - z_t) ≈ 0.88 → **과거를 88% 유지**
- → 60시점 뒤에도 `0.88^60 ≈ 4.9e-4`, gradient 살아남음

이건 3.2에서 배운 forget bias 트릭의 GRU 버전입니다. 실무에선 굳이 건드릴 필요 없이 **기본 초기화로도 충분한 경우가 대부분**이지만, 장기 의존성이 중요한 과제(예: 긴 문서 분류)에선 시도해볼 만합니다.

## 7. LSTM vs GRU — 수식 수준 비교

### LSTM (8개 수식 세트)

```
f_t = σ(W_f · [h_{t-1}, x_t] + b_f)
i_t = σ(W_i · [h_{t-1}, x_t] + b_i)
o_t = σ(W_o · [h_{t-1}, x_t] + b_o)
g_t = tanh(W_g · [h_{t-1}, x_t] + b_g)
c_t = f_t ⊙ c_{t-1} + i_t ⊙ g_t
h_t = o_t ⊙ tanh(c_t)
```

### GRU (4개 수식 세트)

```
r_t = σ(W_r · [h_{t-1}, x_t] + b_r)
z_t = σ(W_z · [h_{t-1}, x_t] + b_z)
h̃_t = tanh(W_h · [r_t ⊙ h_{t-1}, x_t] + b_h)
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t
```

**제거된 것**:
- ❌ Output gate o_t — GRU는 h 자체를 그대로 출력
- ❌ Cell state c_t — h_t로 통합
- ❌ forget vs input의 분리 — update gate z로 통합 (1-z ↔ z)

**유지된 것**:
- ✅ "과거 상태에 대한 선택적 의존" — r_t
- ✅ "볼록 결합식 기억 업데이트" — z_t
- ✅ "덧셈 기반 gradient highway" — (1-z)⊙h_{t-1} 경로
- ✅ sigmoid/tanh의 역할 분담 (게이트 비율 vs 실제 값)

## 8. 파라미터 수 비교 — RNN vs LSTM vs GRU

입력 특징 수 F, hidden 크기 H 일 때:

| 모델 | 가중치 블록 | 파라미터 수 | H=64, F=17 예시 |
|---|---|---|---|
| Vanilla RNN | 1 (W_h, W_x, b) | H² + H·F + H | 4,160 + 1,088 + 64 = **5,312** |
| LSTM | 4 (f, i, g, o) | 4·(H² + H·F + H) | 4 × 5,312 = **21,248** |
| GRU | 3 (r, z, h) | 3·(H² + H·F + H) | 3 × 5,312 = **15,936** |

**관찰**:
- GRU는 LSTM 대비 **25% 파라미터 절감**.
- 작은 데이터셋에선 이게 과적합 방지에 유의미한 차이.
- 우리 프로젝트(MSFT 2310행)처럼 **데이터 양이 제한적**일 때 GRU가 유리.

### Bias 처리의 PyTorch 관례

PyTorch `nn.GRU`는 실제로는 `b_ih`와 `b_hh` 두 개의 bias를 따로 둬서 파라미터가 위의 공식에 H 항이 하나 더 붙습니다 (수학적으로 둘 다 있어도 결과는 하나의 bias와 동일). 이 부분은 실습에서 확인합니다.

## 9. 언제 LSTM을 쓰고 언제 GRU를 쓰는가

**실무 휴리스틱** (Chung et al. 2014, Jozefowicz et al. 2015, 이후 다수 연구):

| 상황 | 권장 | 이유 |
|---|---|---|
| 일반 시계열 / 일반 NLP | **GRU** | 비슷한 성능, 더 적은 파라미터, 더 빠른 학습 |
| 매우 긴 시퀀스 (T > 500) | LSTM 약우위 | 셀 상태 분리가 장기 기억에 유리 |
| 데이터 적음 (< 10만 샘플) | **GRU** | 과적합 위험 감소 |
| 데이터 매우 많음 (> 1M) | 둘 다 괜찮음 | 성능 차이 수렴 |
| 의미적으로 "장기 기억"이 중요 | LSTM | c_t가 장기 기억 전용 파이프로 더 명확 |
| 언어 모델링 | 기본적으론 둘 다 OK | 최근엔 Transformer로 대체됨 |
| 금융 시계열 (우리 프로젝트) | **GRU** | 시퀀스 짧음(T=20~60), 데이터 적음, 과적합 리스크 큼 |

**요약**: **"일단 GRU로 시작, 성능이 안 나오면 LSTM도 시도"** 가 현재 표준 실무 패턴입니다.

## 10. 이 프로젝트가 GRU를 선택하는 7가지 이유

현재 Black-Litterman 프로젝트(MSFT 등 단일 종목, `(2310, 17)` → 시퀀스 `(N, T, 17)`) 문맥에서:

### ① 데이터 양이 제한적

- 2310 행 → sliding window T=20 적용 시 약 2,290 샘플
- 파라미터 25% 적은 GRU가 과적합 방지에 유리
- LSTM의 추가 capacity는 이 데이터 스케일에선 **오히려 독**

### ② 시퀀스 길이가 짧다

- ACF 기반으로 T=10~30 정도면 충분 (1.3 자기상관 분석 결과)
- LSTM의 진가가 나타나는 T > 500 영역이 아님
- 짧은 시퀀스에서는 GRU와 LSTM의 성능 차이가 거의 없음

### ③ 학습 속도가 25% 빠르다

- Walk-Forward 검증(4.1)에서 같은 모델을 **수십 번 재학습**해야 함
- 25% 속도 차이가 실제 체감 차이로 누적됨
- 하이퍼파라미터 탐색도 GRU가 유리

### ④ 해석이 더 단순하다

- 게이트 2개 vs 3개 → 디버깅할 때 들여다봐야 할 것이 적음
- `(1-z, z)` 볼록 결합 → 직관적으로 "과거 vs 새 것" 이해 쉬움
- 비분석가(팀원, 리더)에게 설명하기 쉬움

### ⑤ 출력 인터페이스가 LSTM과 동일

- `nn.LSTM`: `(output, (h_n, c_n))` — 2개 상태 관리
- `nn.GRU`: `(output, h_n)` — 1개 상태만 관리
- Walk-Forward 시 직전 상태를 이어받을 때 코드가 더 깔끔

### ⑥ 과거 문헌에서 금융 시계열 GRU 선택 사례 다수

- 주가 예측, 변동성 예측 등에서 GRU가 LSTM보다 **소폭 우위** 또는 동등한 성능 보고 사례 많음
- 파라미터 경제성과 결합하면 GRU가 현실적 표준

### ⑦ XGBoost 교체 맥락에서 "최소 변경 원칙"

- 현재 XGBoost가 **크지 않은 데이터**에서 잘 돌아가던 모델
- 새 모델도 비슷한 크기의 파라미터 공간이 적절 (GRU 15K vs LSTM 21K)
- 베이스라인 재구축 시 불필요한 복잡도 도입 회피

## 11. PyTorch `nn.GRU` API 미리보기

실습(다음 노트북)에서 본격 다루지만, 미리 맛보기:

```python
import torch
import torch.nn as nn

# 정의
gru = nn.GRU(
    input_size=17,     # F — 특징 수 (현재 프로젝트의 17개 피처)
    hidden_size=64,    # H — hidden 크기
    num_layers=1,      # GRU 층 수 (stacked GRU)
    batch_first=True,  # 입력 shape을 (B, T, F)로 받기
    dropout=0.0,       # num_layers > 1 일 때만 유효
)

# 입력
x = torch.randn(32, 20, 17)   # (B=32, T=20, F=17)
# 초기 hidden (optional, 없으면 0으로 자동)
h0 = torch.zeros(1, 32, 64)    # (num_layers, B, H)

# 실행
output, h_n = gru(x, h0)
# output.shape = (32, 20, 64)  — 전체 시점의 hidden
# h_n.shape    = (1, 32, 64)   — 마지막 시점의 hidden
```

### 주의할 관례

- **LSTM**: `gru(x, (h0, c0))`, 반환 `(output, (h_n, c_n))`
- **GRU**: `gru(x, h0)`, 반환 `(output, h_n)` — **튜플 래핑 없음**
- 이 차이로 학습 루프 코드가 미묘하게 다름 (다음 토픽에서 상세히)

### 가중치 내부 구조

```python
for name, param in gru.named_parameters():
    print(name, param.shape)

# weight_ih_l0   ← 3·H × F — 입력→hidden 가중치 [r, z, h̃]
# weight_hh_l0   ← 3·H × H — hidden→hidden 가중치 [r, z, h̃]
# bias_ih_l0     ← 3·H     — 입력 쪽 bias
# bias_hh_l0     ← 3·H     — hidden 쪽 bias
```

**PyTorch는 r, z, h̃ 순서로 가중치를 묶어서 저장합니다.** LSTM이 [i, f, g, o] 순이었던 것과 대응되는 관례입니다.

---

## 부록 A — GRU의 변형들

### Minimal GRU (MGU, 2016)

게이트를 **1개로 줄인** 버전. update gate z만 남기고 reset gate r을 제거.
```
z_t = σ(W_z · [h_{t-1}, x_t])
h̃_t = tanh(W_h · [h_{t-1}, x_t])
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t
```
일부 과제에선 GRU와 비슷한 성능. 파라미터 33% 추가 절감.

### JANET (Westhuizen & Lasenby, 2018)

LSTM에서 forget gate만 남긴 버전. 실험적으로 많은 과제에서 LSTM과 성능 동등 보고. "게이트가 본질적으로 기억을 유지하는 하나면 충분하다"는 주장.

**메시지**: 더 적은 게이트로도 비슷한 성능 → LSTM의 3게이트가 **반드시 필요한 게 아니라는 증거**. GRU 선택을 뒷받침합니다.

---

## 부록 B — "왜 reset gate는 h_{t-1}에 붙고 update gate는 바깥에 붙는가"

수식 (3)과 (4)를 자세히 보면 reset과 update가 붙는 위치가 다릅니다.

```
h̃_t = tanh(W_h · [r_t ⊙ h_{t-1}, x_t] + b_h)    ── r은 tanh 안쪽
h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t           ── z는 tanh 바깥쪽
```

### 왜 이렇게 설계했는가

**두 게이트의 작용 시점이 근본적으로 다르기 때문**입니다.

1. **Reset gate (안쪽)**: "**후보를 만드는 재료 단계**"에서 과거를 차단
   - 예: "이번 문장 후보를 쓸 때 이전 문맥을 반영할까?"
   - tanh 안에 있어야 **비선형 변환 전에** 과거 정보를 조절할 수 있음.

2. **Update gate (바깥쪽)**: "**최종 상태를 결정하는 단계**"에서 과거와 새 것을 혼합
   - 예: "만든 후보를 실제로 얼마나 채택할까?"
   - tanh 바깥에 있어야 **선형 결합**이 돼서 **gradient highway** 효과를 얻음.

### 만약 반대로 설계했다면?

```
(가설) h̃_t = tanh(W_h · [h_{t-1}, x_t] + b_h)
(가설) h_t = (1 − r_t) ⊙ h_{t-1} + r_t ⊙ (z_t ⊙ h̃_t)
```

이렇게 하면:
- 후보 h̃_t가 항상 전체 과거를 반영 → 과거 차단 불가
- 최종 결합이 더 복잡해져 gradient가 곱 구조로 들어감 → highway 사라짐
- 즉 "리셋 기능"과 "장기 기억 기능" **둘 다 잃음**.

### 결론

원 설계의 **안쪽 r + 바깥쪽 z** 구조는 "**과거 차단 가능 + 장기 기억 유지 가능**" 이라는 두 요구를 동시에 만족하는 **유일한 합리적 배치**입니다. 단순해 보이지만 수학적으로 잘 고민된 구조입니다.

---

## 이 토픽에서 배운 것

1. GRU는 LSTM을 **간소화**한 모델 (3게이트 → 2게이트, c/h 분리 → h 통합).
2. 핵심 수식은 **4줄**이고, `h_t = (1-z)·h_{t-1} + z·h̃_t` 볼록 결합이 요점.
3. **Reset gate r**은 "후보 만들 때 과거 참고 비율", **Update gate z**는 "최종 상태에서 새 것 채택 비율".
4. GRU도 `(1-z)` 덕분에 **gradient highway**를 가짐. 필요 시 `b_z`를 음수로 초기화.
5. 파라미터는 LSTM 대비 **25% 적음**. 학습 속도도 비슷한 비율로 빠름.
6. 실무 권장: **GRU 먼저 시도, 안 되면 LSTM**. 특히 데이터 적고 시퀀스 짧은 경우 GRU 유리.
7. 이 프로젝트는 **7가지 근거**로 GRU 선택: 데이터 양, 시퀀스 길이, 학습 속도, 해석성, 인터페이스 단순성, 문헌 근거, 최소 변경 원칙.
8. PyTorch `nn.GRU`는 `(output, h_n)` 반환, 가중치는 `[r, z, h̃]` 순서.
